# Java 初级后端项目从 0 到 1 实操指南

本文档配套项目：`junior-java-company-demo`

项目地址：<https://github.com/Uphomex/junior-java-company-demo>

本项目是一个“员工与考勤管理”后端项目，目标不是做复杂业务，而是让初学者通过一个完整小项目看懂国内 Java 初级开发常见工作内容：写接口、接收参数、校验数据、处理业务逻辑、操作数据库、返回 JSON、处理异常、写 SQL、运行测试、提交 Git、推送 GitHub。

## 1. 你最终会做出什么

这个项目完成了下面这些功能：

- 部门管理：查询部门、新增部门。
- 员工管理：分页查询员工、新增员工、查询详情、修改员工、员工离职。
- 考勤管理：上班打卡、下班打卡、按日期范围查询考勤。
- 统一返回格式：所有接口都返回 `code`、`message`、`data`、`timestamp`。
- 参数校验：手机号、邮箱、必填字段、分页大小等都做校验。
- 统一异常处理：业务错误不会直接抛出一堆 Java 错误栈，而是返回清晰的 JSON。
- 数据库操作：使用 MyBatis 写 SQL，操作部门、员工、考勤三张表。
- 本地数据库：使用 H2 内存数据库，启动项目即可自动建表和插入测试数据。
- MySQL 切换：准备了 MySQL 配置，后续可以切换到真实 MySQL。
- GitHub 发布：把本地项目提交到 Git，并推送到 GitHub 仓库。

## 2. 本项目有没有前端

本项目没有做 Vue、React 或 HTML 页面，它是一个后端接口项目。

初学者要先理解一件事：公司里的“前后端分离项目”通常分成两部分：

- 前端：页面、按钮、表单、列表。常见技术是 Vue、React、Element Plus、Ant Design Vue。
- 后端：接口、业务逻辑、数据库。常见技术是 Java、Spring Boot、MyBatis、MySQL、Redis。

本项目重点学习后端。虽然没有真实页面，但你仍然能用这些方式模拟前端调用后端接口：

- 浏览器地址栏：适合测试简单的 `GET` 请求。
- `curl` 命令：适合测试 `GET`、`POST`、`PUT`、`DELETE`。
- IntelliJ IDEA 的 `.http` 文件：本项目提供了 `docs/api.http`。
- Postman 或 Apifox：公司里很常用，适合可视化测试接口。

前端真正调用后端时，本质也是发送 HTTP 请求。例如新增员工时，前端会发送：

```http
POST /api/employees
Content-Type: application/json

{
  "name": "赵六",
  "phone": "13900000006",
  "email": "zhaoliu@example.com",
  "departmentId": 1,
  "position": "Java开发工程师",
  "hireDate": "2026-05-11",
  "salary": 12000
}
```

后端处理完后返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "id": 4,
    "name": "赵六"
  },
  "timestamp": "2026-05-11T20:30:00"
}
```

所以本项目虽然没有写页面，但已经覆盖了前后端交互中后端最核心的部分。

## 5. 从 0 到 1 创建项目的整体思路

2. 设计数据库表：department、employee、attendance_record。
3. 创建 Spring Boot 项目。
4. 配置 Maven 依赖。
5. 写数据库建表 SQL 和测试数据。
6. 写实体类 entity。
7. 写 Mapper 接口和 XML SQL。
8. 写 Service 业务逻辑。
9. 写 Controller 接口。
10. 写统一返回和统一异常处理。
11. 用 curl 或 api.http 测试接口。
12. 写简单单元测试。
13. Git 提交。
14. 推送到 GitHub。

这个顺序非常接近真实公司中的开发流程。

## 6. 第一步：分析业务

需求可以先用普通话说清楚：

公司里有很多部门，每个员工属于一个部门。员工每天可以上班打卡和下班打卡。管理员可以查看员工列表、添加员工、修改员工资料、让员工离职，也可以查看员工考勤记录。

从这段话里提取出三个核心对象：

- 部门 Department。
- 员工 Employee。
- 考勤记录 AttendanceRecord。

这三个对象通常会变成三张数据库表，也会变成三个 Java 实体类。

## 7. 第二步：设计数据库表

### 7.1 department 部门表

文件位置：

```text
src/main/resources/schema.sql
```

部门表：

```sql
CREATE TABLE department (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    code VARCHAR(32) NOT NULL UNIQUE,
    name VARCHAR(64) NOT NULL,
    status INT NOT NULL DEFAULT 1,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

字段含义：

- `id`：主键，每条数据唯一编号。
- `code`：部门编码，例如 `RD`、`OPS`。
- `name`：部门名称，例如 `研发部`。
- `status`：状态，`1` 表示启用，`0` 表示停用。
- `created_at`：创建时间。
- `updated_at`：更新时间。

### 7.2 employee 员工表

```sql
CREATE TABLE employee (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(40) NOT NULL,
    phone VARCHAR(20) NOT NULL UNIQUE,
    email VARCHAR(120),
    department_id BIGINT NOT NULL,
    position VARCHAR(60) NOT NULL,
    hire_date DATE NOT NULL,
    salary DECIMAL(12, 2) NOT NULL,
    status INT NOT NULL DEFAULT 1,
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT fk_employee_department FOREIGN KEY (department_id) REFERENCES department(id)
);
```

重点知识：

- `phone UNIQUE` 表示手机号不能重复。
- `department_id` 是部门 ID。
- `FOREIGN KEY` 是外键，表示员工必须属于一个已经存在的部门。
- `DECIMAL(12, 2)` 适合保存金额，不建议用 `double` 保存工资。
- `hire_date DATE` 只保存日期，不保存时分秒。

### 7.3 attendance_record 考勤表

```sql
CREATE TABLE attendance_record (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    employee_id BIGINT NOT NULL,
    work_date DATE NOT NULL,
    check_in_time TIMESTAMP,
    check_out_time TIMESTAMP,
    status INT NOT NULL DEFAULT 1,
    note VARCHAR(255),
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    CONSTRAINT uk_attendance_employee_date UNIQUE (employee_id, work_date),
    CONSTRAINT fk_attendance_employee FOREIGN KEY (employee_id) REFERENCES employee(id)
);
```

重点知识：

- `employee_id` 表示是谁打卡。
- `work_date` 表示哪一天。
- `check_in_time` 是上班打卡时间。
- `check_out_time` 是下班打卡时间。
- `UNIQUE (employee_id, work_date)` 表示一个员工一天只能有一条考勤记录。

## 11. 第六步：写实体类 entity

实体类就是 Java 对象，用来承载数据库数据。

例如员工实体类：

```java
public class Employee {

    private Long id;
    private String name;
    private String phone;
    private String email;
    private Long departmentId;
    private String position;
    private LocalDate hireDate;
    private BigDecimal salary;
    private Integer status;
    private LocalDateTime createdAt;
    private LocalDateTime updatedAt;
}
```

初学者要掌握的 Java 类型：

- `Long`：适合保存数据库主键 ID。
- `String`：字符串，例如姓名、手机号、邮箱。
- `LocalDate`：日期，例如入职日期。
- `LocalDateTime`：日期加时间，例如创建时间、打卡时间。
- `BigDecimal`：金额，例如工资。
- `Integer`：整数，例如状态。

为什么不用 `double` 保存工资：

`double` 是浮点数，可能出现精度误差。工资、金额、订单价格这种业务数据应该使用 `BigDecimal`。

实体类里还有 Getter 和 Setter：

```java
public String getName() {
    return name;
}

public void setName(String name) {
    this.name = name;
}
```

作用：

- `getName()`：读取 name。
- `setName(...)`：修改 name。
- `this.name`：当前对象的 name 字段。

## 12. 第七步：写 DTO 接收请求参数

DTO 是 Data Transfer Object，意思是数据传输对象。

新增员工请求：

```java
public class EmployeeCreateRequest {

    @NotBlank(message = "姓名不能为空")
    private String name;

    @NotBlank(message = "手机号不能为空")
    @Pattern(regexp = "^1[3-9]\\d{9}$", message = "手机号格式不正确")
    private String phone;

    @Email(message = "邮箱格式不正确")
    private String email;

    @NotNull(message = "部门不能为空")
    private Long departmentId;
}
```

为什么要有 DTO：

- 前端传来的参数不一定和数据库字段完全一样。
- DTO 可以专门做参数校验。
- 避免前端直接控制实体类里的所有字段。

常用校验注解：

- `@NotBlank`：字符串不能为空，也不能全是空格。
- `@NotNull`：对象不能为空。
- `@Email`：必须是邮箱格式。
- `@Pattern`：必须符合正则表达式。
- `@Size`：限制字符串长度。
- `@DecimalMin`：限制数字最小值。

Controller 中必须配合 `@Valid` 才会生效：

```java
public ApiResponse<Employee> create(@Valid @RequestBody EmployeeCreateRequest request) {
    return ApiResponse.success(employeeService.create(request));
}
```

## 13. 第八步：写 Mapper 接口

Mapper 接口负责声明“我要做哪些数据库操作”。

例如：

```java
public interface EmployeeMapper {

    List<Employee> search(String keyword, Long departmentId, int offset, int pageSize);

    int count(String keyword, Long departmentId);

    Employee findById(Long id);

    Employee findByPhone(String phone);

    int insert(Employee employee);

    int update(Employee employee);

    int disable(Long id);
}
```

重点理解：

- `List<Employee>`：返回员工列表。
- `Employee`：返回单个员工。
- `int`：通常表示影响了几行数据。
- `@Param`：给 SQL 里的参数命名。

## 14. 第九步：写 MyBatis XML SQL

Mapper 接口只声明方法，真正的 SQL 写在 XML 里。

文件：

```text
src/main/resources/mapper/EmployeeMapper.xml
```

查询员工列表：

```xml
<select id="search" resultMap="EmployeeResultMap">
    SELECT
    <include refid="BaseColumns"/>
    FROM employee
    <where>
        status = 1
        <if test="keyword != null and keyword != ''">
            AND (name LIKE CONCAT('%', #{keyword}, '%') OR phone LIKE CONCAT('%', #{keyword}, '%'))
        </if>
        <if test="departmentId != null">
            AND department_id = #{departmentId}
        </if>
    </where>
    ORDER BY id DESC
    LIMIT #{pageSize} OFFSET #{offset}
</select>
```

知识点：

- `<select>`：查询 SQL。
- `id="search"`：必须和 Mapper 接口方法名一致。
- `resultMap`：告诉 MyBatis 数据库字段如何映射到 Java 对象。
- `<where>`：自动拼接 WHERE。
- `<if>`：动态 SQL，参数有值才加条件。
- `#{keyword}`：安全参数占位，避免 SQL 注入。
- `LIMIT` 和 `OFFSET`：分页。

新增员工：

```xml
<insert id="insert" parameterType="com.example.companydemo.entity.Employee" useGeneratedKeys="true" keyProperty="id">
    INSERT INTO employee (name, phone, email, department_id, position, hire_date, salary, status)
    VALUES (#{name}, #{phone}, #{email}, #{departmentId}, #{position}, #{hireDate}, #{salary}, #{status})
</insert>
```

知识点：

- `useGeneratedKeys="true"`：插入后拿到数据库自动生成的 ID。
- `keyProperty="id"`：把生成的 ID 放回 Java 对象的 `id` 字段。

## 15. 第十步：写 Service 业务逻辑

Service 是业务逻辑层。真实公司里，Controller 不应该写太多业务判断，主要业务规则应该放在 Service。

员工新增逻辑：

```java
@Transactional
public Employee create(EmployeeCreateRequest request) {
    departmentService.getActiveDepartment(request.getDepartmentId());
    ensurePhoneNotUsed(request.getPhone(), null);

    Employee employee = new Employee();
    employee.setName(request.getName());
    employee.setPhone(request.getPhone());
    employee.setEmail(request.getEmail());
    employee.setDepartmentId(request.getDepartmentId());
    employee.setPosition(request.getPosition());
    employee.setHireDate(request.getHireDate());
    employee.setSalary(request.getSalary());
    employee.setStatus(EmployeeStatus.ACTIVE.getCode());

    employeeMapper.insert(employee);
    return employee;
}
```

这段代码做了几件事：

1. 检查部门是否存在。
2. 检查手机号是否被占用。
3. 把 DTO 转成实体类 Employee。
4. 设置员工状态为在职。
5. 调用 Mapper 插入数据库。
6. 返回新增后的员工对象。

`@Transactional` 的作用：

它表示这个方法是一个事务。事务的意思是：一组数据库操作要么全部成功，要么全部失败。

例如新增员工时，如果插入员工成功后又发生异常，事务可以回滚，避免数据库出现半成功状态。

## 17. 第十二步：统一返回格式

文件：

```text
src/main/java/com/example/companydemo/common/ApiResponse.java
```

统一返回类：

```java
public class ApiResponse<T> {

    private int code;
    private String message;
    private T data;
    private LocalDateTime timestamp;
}
```

`<T>` 是 Java 泛型。

为什么要用泛型：

因为不同接口的 `data` 类型不同。

- 查询员工详情：`data` 是 `Employee`。
- 查询员工列表：`data` 是 `PageResult<Employee>`。
- 查询部门列表：`data` 是 `List<Department>`。
- 删除员工：`data` 可以是 `null`。

所以用 `ApiResponse<T>` 可以复用同一个返回结构。

成功返回：

```java
public static <T> ApiResponse<T> success(T data) {
    return new ApiResponse<>(0, "success", data);
}
```

失败返回：

```java
public static ApiResponse<Void> error(int code, String message) {
    return new ApiResponse<>(code, message, null);
}
```

## 18. 第十三步：统一异常处理

文件：

```text
src/main/java/com/example/companydemo/common/GlobalExceptionHandler.java
```

业务异常：

```java
throw new BusinessException("手机号已被其他员工使用");
```

统一处理：

```java
@ExceptionHandler(BusinessException.class)
@ResponseStatus(HttpStatus.BAD_REQUEST)
public ApiResponse<Void> handleBusinessException(BusinessException ex) {
    return ApiResponse.error(ex.getCode(), ex.getMessage());
}
```

好处：

- Service 里只需要抛异常。
- Controller 不需要每个接口都写 try-catch。
- 前端收到的错误格式统一。

例如手机号重复时，前端收到：

```json
{
  "code": 400,
  "message": "手机号已被其他员工使用",
  "data": null,
  "timestamp": "2026-05-11T20:30:00"
}
```

## 19. 第十四步：考勤业务逻辑

考勤涉及几个业务规则：

- 员工必须存在且在职。
- 一天只能上班打卡一次。
- 没有上班打卡，不能下班打卡。
- 一天下班打卡只能一次。
- 9:30 之后上班打卡算迟到。
- 18:00 之前下班打卡算早退。

上班打卡逻辑：

```java
public AttendanceRecord checkIn(AttendanceCheckInRequest request) {
    employeeService.getById(request.getEmployeeId());

    LocalDate today = LocalDate.now();
    AttendanceRecord exists = attendanceMapper.findByEmployeeAndDate(request.getEmployeeId(), today);
    if (exists != null) {
        throw new BusinessException("今天已经上班打卡");
    }

    LocalDateTime now = LocalDateTime.now();
    AttendanceRecord record = new AttendanceRecord();
    record.setEmployeeId(request.getEmployeeId());
    record.setWorkDate(today);
    record.setCheckInTime(now);
    record.setStatus(resolveCheckInStatus(now));

    attendanceMapper.insert(record);
    return record;
}
```

用到的日期时间类：

- `LocalDate.now()`：今天日期。
- `LocalDateTime.now()`：当前日期和时间。
- `LocalTime.of(9, 30)`：上午 9:30。
- `toLocalTime()`：从日期时间中取出时间部分。

## 25. Git 基础知识

Git 是版本管理工具。它可以记录代码每一次变化。

你可以把 Git 理解成“代码存档系统”。

常用概念：

- 工作区：你正在编辑的文件。
- 暂存区：准备提交的文件。
- 本地仓库：你电脑上的 Git 提交记录。
- 远程仓库：GitHub 上的仓库。
- commit：一次代码存档。
- branch：分支，常见主分支叫 `main`。
- remote：远程仓库地址，通常叫 `origin`。
- push：把本地提交推送到 GitHub。
- pull：把 GitHub 上的新代码拉到本地。

## 26. GitHub 推送过程讲解

本项目最终推送到了：

```text
https://github.com/Uphomex/junior-java-company-demo.git
```

完整过程如下。

### 26.1 初始化 Git 仓库

进入项目目录：

```powershell
cd E:\junior-java-company-demo
```

初始化：


作用：让当前文件夹变成一个 Git 仓库。

### 26.2 查看当前状态

```powershell
git status
```

作用：查看哪些文件新增、修改、删除了。

### 26.3 添加文件到暂存区

```powershell
git add .
```

作用：把当前目录下的所有文件加入暂存区。

初学者理解：

`git add .` 不是提交，只是告诉 Git：“这些文件我准备提交。”

### 26.4 创建本地提交

```powershell
git commit -m "Add junior Java company demo"
```

作用：创建一次本地版本记录。

提交信息要简短清楚，例如：

```text
Add junior Java company demo
```

### 26.5 创建 GitHub 仓库

在 GitHub 网站上创建仓库：

```text
junior-java-company-demo
```

创建后得到远程地址：

```text
https://github.com/Uphomex/junior-java-company-demo.git
```

### 26.6 绑定远程仓库

```powershell
git remote add origin https://github.com/Uphomex/junior-java-company-demo.git
```

作用：告诉本地 Git，远程仓库地址在哪里。

查看远程地址：

```powershell
git remote -v
```

### 26.7 第一次推送

```powershell
git push -u origin main
```

含义：

- `push`：推送。
- `origin`：远程仓库别名。
- `main`：要推送的分支。
- `-u`：设置本地 `main` 跟踪远程 `origin/main`，以后可以直接用 `git push`。

### 26.8 遇到远程仓库已有 README 的情况

本项目推送时遇到了这个错误：

```text
Updates were rejected because the remote contains work that you do not have locally.
```

意思是：

GitHub 仓库里已经有一个初始提交，本地没有这个提交，所以 Git 不允许直接覆盖。

解决思路：

1. 先抓取远程内容。
2. 合并远程历史。
3. 再推送。

命令：

```powershell
git fetch origin
git merge origin/main --allow-unrelated-histories
git push -u origin main
```

本项目远程只有一个简单 README，本地 README 更完整，所以合并时保留了本地 README。


并且没有其它文件列表，表示本地和远程同步，当前没有未提交修改。

## 27. 公司里真实开发通常怎么做

真实公司里一般不会直接在 `main` 分支开发，而是：

1. 从 `main` 拉最新代码。
2. 创建功能分支。
3. 在功能分支开发。
4. 提交代码。
5. 推送分支到 GitHub 或 GitLab。
6. 创建 Pull Request 或 Merge Request。
7. 同事 Code Review。
8. 测试通过后合并到 `main`。

常见命令：

```powershell
git pull
git checkout -b feature/employee-api
git add .
git commit -m "Add employee APIs"
git push -u origin feature/employee-api
```

## 28. 初学者建议阅读顺序

建议按这个顺序看代码：

1. `schema.sql`：先看数据库表。
2. `data.sql`：看初始化测试数据。
3. `entity`：看 Java 对象和表字段如何对应。
4. `dto`：看前端请求参数如何接收。
5. `mapper` 接口：看 Java 方法声明。
6. `resources/mapper/*.xml`：看 SQL。
7. `service`：看业务逻辑。
8. `controller`：看 HTTP 接口。
9. `common`：看统一返回和异常处理。
10. `docs/api.http`：动手调用接口。

不要从 Controller 一路乱点。先理解数据库，再看代码会清楚很多。


建议全部设置为 UTF-8。

用户名：

```text
sa
```

密码留空。
