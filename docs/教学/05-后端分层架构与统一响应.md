# 05 后端分层架构与统一响应

国内中大厂的 Java 项目几乎都用"分层架构"。理解这个结构，比熟练一两个 API 重要得多。

## 1. 经典分层

```text
[ Browser / App ] ── HTTP ──► [ Controller ] ──► [ Service ] ──► [ Mapper ] ──► [ DB ]
                                   ↑
                                   │
                              DTO / VO 在层间传递
```

每一层的职责：

| 层 | 关键词 | 干什么 |
| --- | --- | --- |
| Controller | "接口层" | 收 HTTP 请求、做参数校验、调 Service、组装 ApiResponse 返回 |
| Service | "业务层" | 真正的业务逻辑、跨表/跨服务的编排、事务边界 |
| Mapper | "持久层" | 写 SQL、和数据库打交道 |
| Entity | "数据模型" | 一个 Java 类对应一张表 |
| DTO | "传输对象" | 接收前端请求 / 给前端返回的数据形态 |
| Common | "通用层" | 统一响应、异常、工具、配置 |

为什么要分层？三个理由：

1. **可读性**：业务大了不至于"一个 Controller 里 800 行"
2. **可测试**：Service 不依赖 HTTP，单测可以脱离 Spring 跑
3. **可替换**：换一个 RPC 入口（比如 Dubbo）、换一个 ORM，影响范围小

## 2. 项目里的目录布局

```text
src/main/java/com/example/companydemo
├── CompanyDemoApplication.java
├── common/             ← 通用：响应、异常、CORS、分页结构
│   ├── ApiResponse.java
│   ├── BusinessException.java
│   ├── CorsConfig.java
│   ├── GlobalExceptionHandler.java
│   └── PageResult.java
├── controller/         ← HTTP 层
├── service/            ← 业务层
├── mapper/             ← MyBatis 接口
├── entity/             ← DB 实体
├── dto/                ← 入参对象
└── support/            ← 枚举、纯工具
```

> 进阶项目里还会出现：
> - `vo/`：给前端返回的对象（View Object）
> - `query/`：复杂查询条件对象
> - `config/`：Spring 配置类
> - `aspect/`：AOP 切面
> - `event/`、`listener/`：领域事件
> - `enums/`、`constant/`：枚举与常量
> 入门阶段不必都引入，但要看得懂别人的代码。

## 3. Entity vs DTO vs VO

| 类别 | 出现位置 | 典型字段 |
| --- | --- | --- |
| Entity | DB ↔ Mapper | 全字段（含 `id`、`createdAt`） |
| DTO | 前端 → 后端 | 只接收前端会传的字段，**带校验注解** |
| VO | 后端 → 前端 | 只暴露前端需要的字段，**剔除敏感字段** |

举例：

```java
// Entity
public class Employee {
    private Long id;
    private String passwordHash;  // 敏感
    private String name;
    private Long departmentId;
    private LocalDateTime createdAt;
}

// DTO（创建员工时前端传过来的）
public class EmployeeCreateRequest {
    @NotBlank private String name;
    @NotNull  private Long departmentId;
    // 没有 id / passwordHash / createdAt
}

// VO（返回给前端展示的）
public class EmployeeVO {
    private Long id;
    private String name;
    private String departmentName;  // 把 departmentId 翻译成名字
    // 没有 passwordHash
}
```

**为什么要分 Entity 和 DTO/VO？** 直接把 Entity 当入参 / 返参确实简单，但有两个大坑：

1. **数据泄露**：Entity 里有 `passwordHash` 之类敏感字段，序列化给前端就泄露了。
2. **字段污染**：前端可以伪造 `id`、`createdAt` 等不应该被外部修改的字段。

本项目为了让初学者代码量少一点，**返回直接用 Entity**。等你熟练后，建议接口外层换成 VO。

## 4. 统一响应 ApiResponse

```java
public class ApiResponse<T> {
    private int code;
    private String message;
    private T data;
    private LocalDateTime timestamp;

    public static <T> ApiResponse<T> success(T data) {
        return new ApiResponse<>(0, "success", data);
    }

    public static <T> ApiResponse<T> error(int code, String message) {
        return new ApiResponse<>(code, message, null);
    }
}
```

约定：

- `code == 0` 表示成功，其他都是业务/系统错误。
- `data` 用泛型，可以是任意类型：单对象、List、PageResult。
- `timestamp` 方便前端排查问题（"我什么时候调的接口"）。

### 错误码分段

实际项目会把 code 按业务模块分段，比如：

| 段 | 含义 |
| --- | --- |
| 0 | 成功 |
| 1000-1999 | 通用错误（参数错误、未登录） |
| 2000-2999 | 用户模块 |
| 3000-3999 | 订单模块 |
| 9999 | 未知系统错误 |

让前端一看 code 就知道问题在哪里。

## 5. 分页结构 PageResult

```java
public class PageResult<T> {
    private long total;
    private int pageNum;
    private int pageSize;
    private List<T> list;
}
```

前后端约定：

- `total`：满足条件的总记录数
- `pageNum`：当前是第几页（从 1 开始）
- `pageSize`：每页大小
- `list`：本页数据

前端用 `el-pagination`，把 `total / pageSize / current-page` 三个变量一绑，分页器就能用。

## 6. 业务异常 BusinessException

```java
public class BusinessException extends RuntimeException {
    private final int code;

    public BusinessException(String message) {
        this(400, message);
    }

    public BusinessException(int code, String message) {
        super(message);
        this.code = code;
    }

    public int getCode() {
        return code;
    }
}
```

继承 `RuntimeException` 而不是 `Exception`，这样 Service 抛错不用每层都 `throws`。

## 7. 全局异常处理 GlobalExceptionHandler

```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    private static final Logger log = LoggerFactory.getLogger(GlobalExceptionHandler.class);

    @ExceptionHandler(BusinessException.class)
    public ApiResponse<Void> handleBiz(BusinessException e) {
        return ApiResponse.error(e.getCode(), e.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ApiResponse<Void> handleValid(MethodArgumentNotValidException e) {
        String msg = e.getBindingResult().getFieldErrors().stream()
                .map(err -> err.getField() + " " + err.getDefaultMessage())
                .collect(Collectors.joining("; "));
        return ApiResponse.error(400, msg);
    }

    @ExceptionHandler(ConstraintViolationException.class)
    public ApiResponse<Void> handleConstraint(ConstraintViolationException e) {
        return ApiResponse.error(400, e.getMessage());
    }

    @ExceptionHandler(Exception.class)
    public ApiResponse<Void> handleAll(Exception e) {
        log.error("未预期异常", e);
        return ApiResponse.error(500, "系统繁忙，请稍后再试");
    }
}
```

**面试常问**：`@RestControllerAdvice` 和 `@ControllerAdvice` 区别？

`@RestControllerAdvice = @ControllerAdvice + @ResponseBody`，返回值会被序列化成 JSON。

## 8. 流程串起来：新增员工

以"新增员工"这条线为例，看一遍每层做什么：

1. **前端**：`POST /api/employees`，请求体是 JSON
   ```json
   { "name": "赵六", "phone": "13900000006", "departmentId": 1, ... }
   ```
2. **Controller 层**：`EmployeeController.create`
   - `@Valid @RequestBody EmployeeCreateRequest request` 反序列化 + 校验
   - 校验失败 → 全局异常处理器返回 400
   - 校验通过 → 调 `employeeService.create(request)`
3. **Service 层**：`EmployeeService.create`
   - 调 `departmentMapper.findById(request.getDepartmentId())` 检查部门
   - 部门不存在 → 抛 `BusinessException("部门不存在")`
   - 把 DTO 数据搬到 `Employee` 实体
   - 调 `employeeMapper.insert(employee)`
   - 返回 `employee`
4. **Mapper 层**：`EmployeeMapper.insert`
   - 执行 INSERT，自增 id 回填到 entity
5. **Controller 层**：把返回值包成 `ApiResponse.success(employee)`，返回 JSON

## 9. 推荐的命名约定

| 类型 | 命名 | 示例 |
| --- | --- | --- |
| Controller | `XxxController` | `EmployeeController` |
| Service | `XxxService` / `XxxServiceImpl` | `EmployeeService` |
| Mapper | `XxxMapper` | `EmployeeMapper` |
| Entity | 名词单数 | `Employee`、`Department` |
| 入参 DTO | `XxxCreateRequest` / `XxxUpdateRequest` / `XxxQuery` | `EmployeeCreateRequest` |
| 出参 VO | `XxxVO` | `EmployeeDetailVO` |
| 枚举 | `XxxStatus`、`XxxType` | `EmployeeStatus`、`AttendanceStatus` |
