# 12 备份到 E 盘与日常工作流

本章解决一个非常具体的需求：

> "我希望项目既在 GitHub 上（云端，永不丢失），同时在我自己 Windows 电脑的 E 盘上也有一份完整副本（本地，断网也能改）。"

## 1. 推荐方案：GitHub 为主，E 盘做镜像

**重要观念**：

- **GitHub 仓库就是你的"主仓库"**，所有提交都先推到 GitHub。
- **E 盘只是一份"克隆"**，从 GitHub 拉下来，更新方式是 `git pull`。
- 这样**唯一的真相源**只有 GitHub，不会出现"两边都改了不知道哪份是新的"的灾难。

```text
[ 你的工作目录(随便哪个盘) ] ──push──► [ GitHub ]
                                          │
                                          └── pull ──► [ E:\junior-java-company-demo ]（本地备份）
```

## 2. 第一次：把 GitHub 仓库 clone 到 E 盘

打开 Git Bash（或 PowerShell）：

```bash
# 切到 E 盘根
cd /e/        # Git Bash 写法
# 或 PowerShell：
# E:
# cd \

# 克隆
git clone https://github.com/你的用户名/junior-java-company-demo.git
```

完成后，`E:\junior-java-company-demo` 就是完整的项目，包含 `.git` 目录（仓库历史）+ 所有源代码。

> 注意：**不要**把项目直接放在 OneDrive / 百度网盘 / 坚果云同步文件夹里。Git 自己的 `.git` 目录与云盘同步会冲突，造成提交损坏。

## 3. 日常工作流的两种模式

### 模式 A：只在主工作目录上写代码（推荐）

- 在 `D:\workspace\junior-java-company-demo`（或随便哪个工作目录）日常编辑、提交、推送
- E 盘那份**只读**，定期 `git pull` 同步

工作流：

```bash
# 在 D 盘工作目录
cd /d/workspace/junior-java-company-demo
# 改代码 ...
git add .
git commit -m "feat: 新增登录页"
git push

# 切到 E 盘备份目录
cd /e/junior-java-company-demo
git pull
```

### 模式 B：在 E 盘也直接写代码（也行）

如果你**就是想直接在 E 盘改**：

```bash
cd /e/junior-java-company-demo
# 改代码
git add .
git commit -m "feat: 修复手机号校验"
git push
```

然后**没有"另一份"了**，因为 E 盘那份既是工作目录又是备份。这种模式下"E 盘备份"和"GitHub 备份"是一体的。

## 4. 写一个自动备份脚本（可选）

### Windows：PowerShell 一键拉取

`E:\backup_repos.ps1`：

```powershell
$repos = @(
  'E:\junior-java-company-demo'
)
foreach ($repo in $repos) {
  Write-Host "=== Pulling $repo ===" -ForegroundColor Cyan
  Set-Location $repo
  git pull
}
```

双击运行，或者用任务计划程序设成"每天 22:00 自动跑"。

### Windows：批处理 .bat

`E:\backup_repos.bat`：

```bat
@echo off
cd /d E:\junior-java-company-demo
echo === Pulling junior-java-company-demo ===
git pull
pause
```

### macOS / Linux：shell 脚本

`~/backup_repos.sh`：

```bash
#!/usr/bin/env bash
set -e

REPOS=(
  "$HOME/backup/junior-java-company-demo"
)

for repo in "${REPOS[@]}"; do
  echo "=== Pulling $repo ==="
  cd "$repo"
  git pull
done
```

```bash
chmod +x ~/backup_repos.sh
~/backup_repos.sh
```

## 5. 把 E 盘备份"完全只读"（避免误改）

如果你怕在 E 盘那份上不小心 commit / push 出意外：

```bash
cd /e/junior-java-company-demo
git config core.fileMode false      # Windows 上避免权限位混乱
# 不必把它锁成只读，定期 git pull 就行
```

或者，干脆**只保留 `.git`** 用于"裸备份"：

```bash
# 在 E 盘建一个 mirror 镜像（不带工作区）
git clone --mirror https://github.com/你的用户名/junior-java-company-demo.git E:\backup\junior-java-company-demo.git

# 定时更新
cd /e/backup/junior-java-company-demo.git
git remote update
```

这种 mirror 模式备份的是**完整的仓库历史**，体积小，最适合"灾备"。

## 6. 多地点同步整理

| 位置 | 角色 | 操作 |
| --- | --- | --- |
| 公司电脑 D 盘 | 主工作目录 | 每天写代码、`add/commit/push` |
| 家里电脑 D 盘 | 第二工作目录 | 来了先 `git pull`，工作后 `git push` |
| 家里电脑 E 盘 | 本地备份 | `git pull` 定期同步，不改 |
| GitHub | 云端权威 | 默认目的地，所有改动最终都在 |

**记住**：GitHub 是唯一权威。其他位置都是"同步副本"，永远靠 `git pull` 跟它对齐。

## 7. 不止备份代码：备份数据库

代码可以 git，**数据库数据 git 不了**（也不应该 git）。如果你的项目切到了 MySQL：

```bash
# 备份
mysqldump -uroot -p company_demo > E:\backup\company_demo_2026-05-13.sql

# 恢复
mysql -uroot -p company_demo < E:\backup\company_demo_2026-05-13.sql
```

定时备份：Windows 任务计划程序里跑这条命令，每天自动备份到 E 盘的 `backup\` 下，按日期命名。

## 8. 不止备份代码：备份资产

如果项目里有上传图片、附件等，应该放到 OSS / S3 / 七牛云之类的对象存储里，不要存在数据库或服务器磁盘上。理由：

- 服务器磁盘坏了数据就丢了
- 对象存储有冗余备份，比自己搞稳

## 9. 灾难恢复演练（强烈推荐做一次！）

设想：**笔记本被偷了，怎么 5 分钟内恢复开发？**

1. 新机器装 Git（[01-环境准备](./01-环境准备与工具安装.md)）
2. `git clone https://github.com/你的用户名/junior-java-company-demo.git`
3. 装 JDK、Maven、Node、pnpm
4. `mvn spring-boot:run` & `pnpm install && pnpm dev`
5. 继续干活

如果这五步走不通，说明你的**项目状态不全在 GitHub 上**，缺什么就补什么。这才是"备份"的意义。

## 10. 安全提示

- **不要把密码、私钥、`application-prod.yml` 提交到 GitHub**。哪怕仓库是私有的，也不要。
- 如果不小心提交了 → **立刻去对应平台注销那个密钥/token**，重新生成。Git 历史里清掉是次要的。
- `.gitignore` 是你的护栏，commit 前用 `git status` 看一眼，不该上的别上。

## 11. 一句话总结

> **代码进 GitHub，备份回 E 盘，数据库另备份，资产存对象存储。**
